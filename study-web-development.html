포트폴리오 웹사이트 제작기
문제를 구조적으로 해석하며 완성한 첫 웹 개발 기록

빠르게 만드는 것보다 중요한 것은, 왜 문제가 발생했는지 이해하는 것이었습니다.

이 기록은 포트폴리오 웹사이트를 제작하며 마주한 CSS 충돌, 레이아웃 구조 문제, 클래스 네이밍 충돌, JavaScript 인터랙션 구현, 배포 후 캐시 이슈까지 실제 문제를 분석하고 해결한 과정을 정리한 학습 기록입니다.

데이터 분석에서도 저는 같은 방식으로 접근합니다. 겉으로 보이는 현상만 고치는 것이 아니라, 원인을 분해하고 구조를 확인한 뒤 해결하는 과정을 중요하게 생각합니다.

1. 시작점: 빨리 만드는 것과 제대로 만드는 것은 달랐다

처음 포트폴리오 웹사이트를 만들 때의 목표는 단순했습니다. HTML과 Tailwind CSS를 활용해 빠르게 결과물을 완성하는 것이었습니다. 화면에 보이는 카드와 버튼, 레이아웃 정도는 비교적 쉽게 구현할 수 있을 것이라 생각했습니다.

하지만 실제 작업은 예상과 달랐습니다. 스타일이 분명히 적용되어 있어야 하는데 반영되지 않았고, 레이아웃은 의도와 다르게 깨졌으며, 배포 후에는 수정한 코드가 브라우저에서 바로 보이지 않는 문제도 반복되었습니다.

이 과정에서 가장 크게 느낀 점은, 대부분의 문제는 새로운 기술이 부족해서가 아니라 기본 동작 원리를 정확히 이해하지 못했기 때문에 발생한다는 것이었습니다. 그래서 단순히 “되게 만드는 것”이 아니라, 왜 이런 현상이 발생했는지 구조적으로 분석하는 방식으로 접근했습니다.

2. 문제 해결 과정: 구현보다 원인 파악에 집중하다

2-1. CSS 충돌과 우선순위: 스타일이 적용되지 않았던 이유

문제 상황

Tailwind CSS를 활용해 레이아웃을 구성하던 중, h-full 같은 높이 관련 클래스가 정상적으로 적용되지 않는 문제가 발생했습니다. 클래스는 분명히 들어가 있었지만 화면에서는 기대한 결과가 나타나지 않았습니다.

원인 분석

처음에는 단순히 클래스를 잘못 쓴 것이라고 생각해 여러 유틸리티 클래스를 바꿔보았습니다. 하지만 문제가 반복되어 브라우저 개발자 도구의 Elements / Computed 탭으로 최종 적용 스타일을 확인했습니다.

확인 결과, Tailwind 클래스가 적용되지 않은 것이 아니라 다른 CSS 선언에 의해 덮어씌워지고 있던 것이 원인이었습니다. 이 과정에서 CSS 우선순위가 실제 화면 결과에 얼마나 직접적으로 영향을 미치는지 체감했습니다.

  • 인라인 스타일, ID, 클래스, 태그는 각각 적용 우선순위가 다름
  • Tailwind 유틸리티 클래스도 이 우선순위 체계 안에서 동작함
  • 보이는 문제는 단순 스타일 문제가 아니라 “최종 계산 결과” 문제일 수 있음

해결 방법

이후 모든 것을 Tailwind만으로 해결하려는 방식에서 벗어나, style.css에서 필요한 스타일을 명확히 재정의하는 방식으로 구조를 정리했습니다.

특히 !important를 남발하는 대신 더 구체적인 선택자를 사용해 우선순위를 자연스럽게 확보했습니다. 그 결과 Tailwind의 생산성은 유지하면서도 세부 디자인은 안정적으로 제어할 수 있었습니다.

2-2. Flexbox 레이아웃: CSS보다 먼저 봐야 할 것은 HTML 구조였다

문제 상황

프로젝트 카드들을 균일한 형태로 배치하고 싶었지만, display: flex를 적용했음에도 카드 정렬이 일정하지 않았습니다. 처음에는 gap, width, align-items 같은 CSS 속성만 계속 수정했습니다.

원인 분석

문제를 해결하기 위해 개발자 도구의 Elements 탭에서 실제 DOM 구조를 다시 확인했습니다. 그 결과, 중간에 불필요한 wrapper div가 들어가 있어 flex 컨테이너의 직계 자식이 카드가 아니라 래퍼 요소가 되어 있었습니다.

이 경험을 통해 Flexbox는 단순히 속성을 주는 것만으로 해결되는 것이 아니라, 어떤 요소가 flex 컨테이너이고 어떤 요소가 직계 자식인지가 핵심이라는 점을 분명히 이해하게 되었습니다.

해결 방법

불필요한 wrapper를 제거하고 HTML 구조를 단순화했습니다. 그러자 별도의 복잡한 CSS 수정 없이도 카드 레이아웃이 안정적으로 정렬되기 시작했습니다.

이 과정은 레이아웃 문제의 상당수는 CSS 속성보다 HTML 구조에서 시작된다는 점을 체감하게 만든 대표적인 사례였습니다.

2-3. 클래스 네이밍 충돌: 이름이 모호하면 구조도 흔들린다

문제 상황

포트폴리오 목록 페이지에서는 잘 보이던 스타일이 상세 페이지에서 전혀 다른 방식으로 적용되는 문제가 있었습니다. 분명 비슷한 카드 UI였지만, 실제 동작은 일관되지 않았습니다.

원인 분석

확인 결과, 서로 다른 페이지에서 같은 클래스명을 다른 용도로 사용하고 있었습니다. 예를 들어 목록 카드와 상세 페이지 컨테이너 모두에 비슷한 의미의 클래스명이 사용되면서 CSS가 예상하지 못한 방식으로 섞이고 있었습니다.

즉 문제의 본질은 스타일 자체보다 컴포넌트 역할이 이름에 명확히 반영되지 않았다는 점이었습니다.

해결 방법

클래스명을 더 구체적으로 재설계했습니다. 예를 들어 상세 페이지의 주요 컨테이너는 .project-detail-container처럼 역할이 드러나는 이름으로 변경했습니다.

이 경험을 통해 클래스 네이밍은 단순한 표기가 아니라, 구조를 설명하고 충돌을 예방하는 설계 도구라는 점을 배웠습니다.

2-4. JavaScript 모달 구현: 복사에서 이해로 넘어간 첫 경험

문제 상황

프로젝트 이미지를 클릭했을 때 확대해서 볼 수 있는 기능을 구현하고 싶었습니다. 처음에는 CSS만으로 해결할 수 있을 것이라 생각했지만, 사용자 이벤트에 따라 상태가 바뀌는 기능은 JavaScript가 필요했습니다.

접근 방식

처음에는 예제 코드를 참고해 동작만 맞추려 했습니다. 하지만 코드가 왜 동작하는지 이해하지 못하니, 조금만 수정해도 전체 흐름이 흔들렸습니다.

그래서 단순 복사 대신 아래 개념부터 다시 정리했습니다.

  • DOM이 무엇인지
  • querySelector가 요소를 어떻게 선택하는지
  • onclick, addEventListener가 어떤 방식으로 이벤트를 처리하는지
  • 현재 선택된 이미지를 어떻게 추적하고 넘겨줄 수 있는지

구현 결과

최종적으로는 이미지 클릭 시 모달 열기, 이전/다음 이미지 이동, 바깥 영역 클릭 시 닫기까지 하나의 흐름으로 구현할 수 있었습니다.

이 경험은 단순히 기능 하나를 만든 것이 아니라, 웹페이지가 사용자 이벤트에 반응하는 구조를 처음으로 직접 이해한 과정이었습니다.

2-5. 데이터 시각화 적용: 차트는 장식이 아니라 메시지 전달 도구였다

문제 상황

Instacart 프로젝트의 분석 결과를 웹 포트폴리오에 옮기면서, 숫자와 표만으로는 핵심 메시지를 빠르게 전달하기 어렵다는 한계를 느꼈습니다.

접근 방식

그래서 Chart.js를 활용해 고객 세그먼트 분포와 주요 결과를 시각화했습니다. 단순히 예쁜 그래프를 넣는 것이 목표가 아니라, 사용자가 한눈에 핵심 패턴을 이해할 수 있도록 하는 것에 집중했습니다.

배운 점

이 과정에서 가장 크게 느낀 점은, 시각화는 “무엇을 보여줄 것인가”가 먼저라는 점이었습니다. 같은 데이터라도 차트 유형과 강조 포인트에 따라 전달력은 크게 달라졌습니다.

결국 시각화는 디자인 요소가 아니라 분석 결과를 더 빠르게 이해시키는 커뮤니케이션 도구라는 점을 명확히 배웠습니다.

2-6. 카드 높이와 UI 일관성: 보기 좋은 정렬은 자동으로 만들어지지 않았다

문제 상황

카드형 UI를 구성하면서 카드마다 텍스트 길이가 달라 높이가 제각각으로 보였습니다. 기능적으로는 문제가 없었지만, 포트폴리오 전체 인상에서는 완성도를 떨어뜨리는 요소였습니다.

접근 방식

단순히 높이를 고정하는 방식보다는, 내용량이 달라도 자연스럽게 균형을 맞출 수 있는 구조를 고민했습니다.

  • min-height를 사용해 최소 높이 기준 확보
  • Flexbox 정렬을 활용해 내부 요소 배치 일관성 유지
  • 태그, 버튼, 설명 영역이 같은 흐름에서 정렬되도록 구조 정리

배운 점

사용자는 보통 “깔끔하다”는 결과만 보지만, 그 결과 뒤에는 작은 정렬 기준과 레이아웃 설계가 쌓여 있다는 것을 알게 되었습니다. 즉 UI 완성도는 큰 기능보다 작은 일관성의 합에서 나온다는 점을 체감했습니다.

2-7. 배포 후 캐시 문제: 코드가 틀린 게 아니라 브라우저가 예전 파일을 보여주고 있었다

문제 상황

GitHub Pages로 사이트를 배포한 뒤 코드를 수정하고 푸시했는데, 브라우저에서는 이전 버전이 계속 보이는 문제가 발생했습니다. 저장소에는 최신 코드가 올라가 있었기 때문에 처음에는 배포 자체를 의심했습니다.

원인 분석

원인은 브라우저 캐시였습니다. 브라우저가 CSS, JavaScript, 이미지 파일을 저장해두고 재사용하면서 실제 서버의 최신 파일 대신 로컬 캐시 파일을 보여주고 있었습니다.

해결 방법

강력 새로고침을 통해 캐시를 무시하고 새 파일을 불러오자 정상적으로 반영되었습니다. 이후부터는 배포 문제를 확인할 때 코드 → 배포 → 캐시 순서로 점검하는 습관을 가지게 되었습니다.

이 경험은 내 환경에서 보이는 것과 실제 사용자 환경에서 보이는 것은 다를 수 있다는 점을 실감하게 했습니다.

3. 핵심 인사이트: 기본기를 이해하면 문제를 빠르게 줄일 수 있다

  • 문제의 상당수는 새로운 기술 부족이 아니라 기본 원리 이해 부족에서 발생한다.
  • 스타일 문제가 생기면 CSS만 보기보다 HTML 구조와 적용 우선순위를 함께 봐야 한다.
  • 개발자 도구는 화면 문제를 감으로 수정하는 대신 원인을 직접 확인하게 해주는 핵심 도구다.
  • 문제를 “해결했다”보다 “원인을 설명할 수 있다” 수준까지 가야 비슷한 문제를 다시 줄일 수 있다.

이번 작업은 단순히 웹사이트 한 개를 만든 경험이 아니라, 문제를 구조적으로 해석하고 빠르게 원인을 좁혀가는 방식의 중요성을 배우는 과정이었습니다.

이는 이후 데이터 프로젝트에서도 그대로 이어졌습니다. 지표 변화라는 결과만 보는 것이 아니라, 어떤 구조가 그 결과를 만들었는지 해석하고 다시 실행안으로 연결하는 사고방식의 기반이 되었습니다.

사용한 기술 스택
도구 / 기술 역할
HTML / CSS 기본 구조 설계 및 레이아웃 구성

4. 마무리

포트폴리오 웹사이트를 만드는 과정에서 가장 크게 배운 것은 화려한 기술보다 기본기를 바탕으로 문제를 해석하는 힘이 더 중요하다는 점이었습니다.

앞으로 새로운 기술을 배우더라도 먼저 이 기술이 어떤 구조로 동작하는지, 문제가 생기면 어디서부터 확인해야 하는지를 먼저 생각하는 방식으로 접근하고자 합니다.

이 기록은 웹 개발 입문 경험이면서 동시에, 작은 문제를 빠르게 포착하고 원인 중심으로 해결해 나가는 문제 해결 방식의 출발점이기도 합니다.
